# 文件和文件夹创建及管理规范

## 1. 概述

本规范旨在为 OpenClaw `workspace` 下的文件和文件夹创建、命名、存放、管理提供一套清晰的指导原则。遵循本规范有助于提高工作效率，确保文件组织结构清晰，便于查找、维护和协作。

## 2. 核心原则

*   **清晰性 (Clarity)**：文件名和文件夹名应清晰、简洁、具有描述性，避免使用模糊不清的名称。
*   **一致性 (Consistency)**：在整个 `workspace` 中，文件和文件夹的命名、结构应保持一致性。
*   **可查找性 (Findability)**：文件应存放在逻辑上最合理的目录中，方便快速定位。
*   **版本控制 (Version Control)**：重要文件应考虑纳入版本控制，尤其是在协作环境中。
*   **绝对路径 (Absolute Paths)**：在涉及跨目录操作或子代理任务时，**强制使用绝对路径**以避免路径解析错误。

## 3. 规范细则

### 3.1 文件夹结构

*   **`workspace` 根目录**：
    *   **`/projects`**：存放所有项目相关的代码、数据、文档等。每个项目应有独立的子文件夹。
        *   示例：`/projects/stock-data-center`、`/projects/fox-skills`
    *   **`/docs`**：存放通用文档、技能指南、规范手册等。
        *   示例：`/docs/skills_guides`、`/docs/project_docs`
    *   **`/memory`**：存放代理的每日笔记和长期记忆文件。
        *   示例：`/memory/2026-02-12.md`、`/MEMORY.md`
    *   **`/skills`**：存放自定义技能的定义文件。
        *   示例：`/skills/my_custom_skill`
    *   **`/temp`**：用于存放临时的、无需长期保留的文件和脚本。**此文件夹下的内容应定期清理。**
        *   示例：`/temp/downloads`、`/temp/scripts`

### 3.2 命名规范

*   **文件名和文件夹名**：
    *   **小写字母 + 下划线**：推荐使用小写字母和下划线 (`_`) 进行分隔，例如 `my_new_project`、`data_analysis_report.md`。
    *   **避免特殊字符**：避免使用空格、中文、特殊符号（如 `!@#$%^&*()+=[]{}|;:'",.<>?`）。
    *   **日期格式**：如果文件名需要包含日期，推荐使用 `YYYYMMDD` 或 `YYYY-MM-DD` 格式，例如 `report_20260212.md`。
    *   **版本号**：如果文件需要版本控制（非 Git），可以在文件名中添加版本号，例如 `script_v1.py`。

### 3.3 文件类型存放

*   **代码文件 (`.py`, `.js`, `.sh`, `.json`, `.yaml` 等)**：
    *   **项目代码**：应存放在各自项目的 `/projects/<project_name>/` 目录下，并根据功能进一步细分文件夹（例如 `/projects/<project_name>/src`、`/projects/<project_name>/utils`）。
    *   **技能代码**：应存放在 `/skills/<skill_name>/` 目录下。
    *   **临时脚本**：不属于任何特定项目的临时脚本，应存放在 `/temp/scripts` 目录下，并在使用完毕后及时清理。
*   **文档文件 (`.md`, `.docx`, `.pdf` 等)**：
    *   **项目文档**：应存放在各自项目的 `/projects/<project_name>/docs` 目录下。
    *   **通用指南/规范**：应存放在 `/docs/` 目录下。
*   **数据文件 (`.csv`, `.xlsx`, `.db` 等)**：
    *   应存放在各自项目的 `/projects/<project_name>/data` 目录下。
*   **日志文件 (`.log`)**：
    *   应存放在 `/logs` 或 `/projects/<project_name>/logs` 目录下。
*   **下载文件 (`.zip`, `.rar`, 临时图片等)**：
    *   **临时下载**：应首先存放在 `/temp/downloads` 目录下。
    *   **归档**：如果需要长期保留，应根据其类型和用途，移动到 `/projects/<project_name>/data` 或 `/docs` 等合适的归档目录中，并进行重命名和分类。
    *   **清理**：`/temp/downloads` 目录应定期清理。

### 3.4 权限管理

*   **最小权限原则**：除非明确需要，否则文件和文件夹应遵循最小权限原则，避免赋予不必要的读写执行权限。
*   **敏感信息**：包含敏感信息（如 API Key、密码）的文件，应严格限制访问权限，并考虑使用环境变量或密钥管理服务，而不是直接硬编码在文件中。

## 4. 特殊文件处理

*   **`.env` 文件**：用于存放环境变量，应在 `.gitignore` 中忽略，确保不提交到版本控制。
*   **`README.md` 文件**：每个项目、每个重要的子模块都应包含一个 `README.md` 文件，简要介绍其功能、使用方法和注意事项。

## 5. 执行流程

1.  **确定用途和类型**：在创建任何文件或文件夹之前，明确其用途和文件类型。
2.  **选择存放位置**：根据文件类型和用途，选择最合适的存放路径。
3.  **遵循命名规范**：为文件或文件夹选择清晰、一致的名称。
4.  **使用绝对路径**：在代码或指令中引用文件路径时，始终使用绝对路径。
5.  **定期清理**：定期检查 `/temp` 目录和下载文件夹，清理不再需要的文件。
