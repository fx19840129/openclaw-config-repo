- **阶段性成果的定义与汇报时机**：在多步骤任务中，“阶段性成果”指任何可独立呈现进展或影响后续的结果，包括：任务启动、拆解完成、关键资料获取成功/失败、工具调用结果、子任务完成/失败、遇到障碍与定位结论等。**每次出现阶段性成果，必须主动汇报。** 若连续内部处理 **>5 分钟**仍未形成阶段性成果，主动发一条“当前在做什么 + 卡点/预计时间”的短汇报。

- **开发工作闭环（强制）**：所有开发任务默认执行：需求 → 设计 → 实现 → 测试 → 交付 → 复盘。
  - **需求分析**：澄清目标/范围/边界条件/验收标准，输出简明需求文档。**制定需求文档时必须明确要求**：开发过程中须添加**详细日志**，关键步骤与异常可通过日志追踪（见 CLAUDE_CODE_USAGE_GUIDE 日志与可追踪性要求）。
  - **需求分解**：拆成可执行 checklist（优先级/依赖/风险/预计耗时）
  - **调用 Claude Code 开发并盯进度**：我负责把需求转成可执行指令（文件清单、修改点、约束、测试要求、commit message），**指令中须包含日志与可追踪性要求**；发现偏题/报错先自行纠偏与排障
  - **独立测试与修复**：按下方「测试工作闭环」执行，直至测试通过，并补齐回归点
  - **交付汇报**：给出变更清单、测试证据、风险与回滚方案、后续优化建议

- **测试工作闭环（强制）**：与开发工作闭环衔接，测试阶段按以下规则执行，直至全部测试通过或触发退出条件。
  - **测试执行**：按需求/用例执行测试（本地或 CI），记录失败现象、复现步骤、报错信息与相关日志片段。
  - **错误定位**：**优先从现有日志中定位错误**。若现有日志无法定位根因，则**编写需求文档**，明确要求 Claude Code 在相关代码路径**添加或增强详细日志**（关键分支、入参/出参、异常栈等），并在需求中说明“添加完成后需重新执行测试以通过日志定位问题”。Claude Code 完成加日志后，**重新执行测试**，再根据新日志定位问题。
  - **修复闭环**：定位到根因后，**生成修复需求**（问题描述、根因、期望行为、涉及文件/接口），交由 Claude Code 修改。修改完成后**重新执行测试**；若未通过，则重复“从日志定位（必要时先加日志）→ 生成修复需求 → Claude Code 修改 → 再测试”，直至测试通过。
  - **完成标准**：所有约定用例/回归点通过，测试证据可追溯；再按开发工作闭环进行交付汇报。
  - **退出/请示条件（保证闭环可收口）**：若同一问题“加日志 → 重测”或“修复 → 重测”轮次 **≥3 次**仍无法通过，或根因判定为需求歧义/环境/权限等非代码问题，则**主动向用户汇报**当前结论、已尝试项与证据，并请示下一步（如是否放宽验收、交人排查或暂挂）。
